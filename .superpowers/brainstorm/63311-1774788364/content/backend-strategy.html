The "Invisible" Backend: Architecture & Deployment

How the Bot and Dashboard share a single database and logic without a third server.

1. Monorepo Structure (Code Location)

2. Where the "Backend" Lives

Integrated API Strategy

We use Next.js API Routes inside apps/dashboard.

// apps/dashboard/api/config.ts
export default async function handler(req, res) {
  const { data } = await supabase.from('configs')...;
  res.status(200).json(data);
}

✓ No separate backend server needed. Vercel runs these as serverless functions.

Why this is better

  • One less server: You don't have to pay for or manage a third "Backend" instance.
  • Shared Types: The Bot and Dashboard use the exact same database definitions from packages/db.
  • Security: Supabase's built-in "Row Level Security" ensures the Bot can only see what it needs.

The Trade-off

  • Communication: If the Dashboard needs to "tell" the Bot something instantly, we use Supabase Realtime (WebSockets) instead of an internal API call.
